# Codex 题组提问需求

## 背景

Chill Vibe 已能把 Codex 输出的单个 `<ask-user-question>` XML 块渲染为可点击的提问卡片，但注入给 Codex 的协议只允许一次询问一个问题。遇到需要同时确认多个互不依赖的信息时，Codex 只能逐题往返，或退化成普通文本，影响效率与稳定渲染。

Claude 原生 `AskUserQuestion` 活动和前端提问卡片已经支持 `questions[]` 题组，因此本次只补齐 Codex 合成协议与解析链路，不重新设计整套交互。

## 需求

1. Codex 在确实需要用户决定后才能继续时，可以在一个 `<ask-user-question>` XML 块内发送一组问题。
2. 题组协议使用顶层 `questions` 数组；每一题包含：
   - `header`：简短标题。
   - `question`：一句清晰问题。
   - `multiSelect`：当前保持 `false`。
   - `options`：2-3 个带简短说明的选项。
3. 题组数量不设上限；只要问题适合一起回答，Codex 可以在同一个 `questions[]` 中完整发送。只有一个问题时仍可使用现有单题格式。
4. Codex 结构化输出解析器必须同时兼容：
   - 现有顶层 `question/options` 单题格式。
   - 新增顶层 `questions[]` 题组格式。
5. 题组必须复用现有提问卡片，允许逐题切换、全部回答后一次提交，并把所有答案作为一次后续用户消息发回 Codex。
6. 非法题目不能造成整条消息崩溃：解析时忽略缺少问题文本或有效选项的条目；只要仍有至少一题有效，就渲染有效题目。
7. 英文与简体中文的 Codex 系统指令都必须描述相同的题组能力与限制。

## 非目标

- 不在本次新增题内多选 UI；`multiSelect` 仍固定为 `false`。
- 不改变 Claude 原生 `AskUserQuestion` 行为。
- 不允许 XML 块前后混入普通回复文本；Codex 仍需只输出一个完整块并等待用户回答。
- 不新增持久化字段或迁移逻辑。
