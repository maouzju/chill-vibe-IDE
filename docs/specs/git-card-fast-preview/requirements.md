# Git 卡牌快显加载需求

## 背景

Git 卡牌打开时会先等待完整 `inspectGitWorkspace()` 返回。完整状态会读取 diff 预览、最近提交和仓库描述；在改动多、文件大或磁盘较慢时，卡牌只显示“正在加载本地改动”，用户需要等到全部加载完成才看到「提交新增」「分析改动」「古法 Git」。

用户反馈：即使完整明细还没加载完，应用也应该已经能知道大概有多少改动，并尽早给出这几个操作入口。

## 目标

1. Git 卡牌首次打开时必须优先获取轻量状态：仓库是否存在、分支、ahead/behind、改动文件列表和数量。
2. 轻量状态可用后，即使完整 diff/最近提交仍在加载，卡牌也要展示改动数量和主要操作按钮：提交新增、分析改动、古法 Git。
3. 操作必须安全：点击提交新增、分析改动、古法 Git 时应基于最新可用状态继续执行；必要时由现有流程再拉取完整状态。
4. 完整状态加载完成后，文件列表和统计要自然升级，不闪成空状态，不隐藏已显示的按钮。
5. 非 Git 仓库、无工作区、错误态保持现有引导和刷新能力。
6. 轻量状态不能包含大 patch 文本，避免把慢加载和大 payload 提前搬到首屏。

## 非目标

- 不重做完整 Git 界面。
- 不引入后台常驻 watcher。
- 不改变提交、暂存、同步的语义。
- 不要求在轻量状态中提供准确增删行数；没有 patch 前可显示 `+0 / -0` 或弱化统计。

## 验证

- 后端单元测试覆盖轻量状态不带 patch 且符合共享 schema。
- 前端/桥接测试覆盖新 API 能从 Electron bridge 获取并校验。
- Git 卡牌行为测试覆盖：完整状态延迟时，轻量状态到达后按钮和改动数已可见。
