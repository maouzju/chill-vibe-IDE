# Git 回退改动（Discard Changes）— Requirements

## 背景

「古法 Git」面板（`GitFullDialog`）目前支持查看 diff、暂存、提交、拉取、推送和历史，但没有像 GitHub Desktop 那样的 "Discard changes" 能力：用户无法在面板里把某个文件（或一批文件）的本地改动一键回退到 HEAD 状态。

## 用户故事

- 作为用户，我在古法 Git 面板选中一个改动文件后，可以点「回退改动」把这个文件恢复到最近一次提交的状态。
- 作为用户，我可以一次回退当前列表里所有勾选/可见的改动。
- 作为用户，回退是破坏性操作，执行前必须有明确的确认步骤，避免误触。

## 验收标准

1. **tracked 修改**：对已跟踪且被修改的文件执行回退后，文件内容恢复为 HEAD 版本，`git status` 不再显示该文件。
2. **staged 改动**：已暂存（staged）的修改同样被回退——暂存区和工作区一起恢复到 HEAD。
3. **untracked / 新增文件**：回退等于删除该文件（含 staged 的新增 `A` 状态：先移出索引再删除文件）。
4. **删除的文件**：工作区里被删除的 tracked 文件回退后被恢复。
5. **重命名**：`R` 状态回退后恢复原路径文件、删除新路径文件。
6. **冲突文件**：冲突（conflicted）文件不允许通过此入口回退，UI 上禁用。
7. **大批量路径**：几百个路径回退不会触发 Windows `spawn ENAMETOOLONG`（pathspec 走 stdin，见 pitfall #174）。
8. **确认交互**：单文件回退与全部回退都需要一次显式确认；确认 UI 在 light/dark 两主题下均正常。
9. **状态刷新**：回退成功后面板状态自动刷新，回退的文件从变更列表消失；失败时显示错误 notice 并刷新状态。
10. **双语**：所有新增文案提供 zh-CN 与 en。

## 非目标

- 不做回收站/撤销回退（GitHub Desktop 的 trash 恢复不在本期）。
- 不做单个 hunk/行级回退。
- 不改 GitToolCard 紧凑视图，仅古法 Git 全量面板。
