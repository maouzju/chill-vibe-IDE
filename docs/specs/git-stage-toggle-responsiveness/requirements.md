# 古法 Git 暂存切换响应需求

## 背景

古法 Git 在约 120 个改动文件的仓库中切换单文件暂存复选框时，现有 Electron 性能门禁记录到
数秒级耗时。不过旧门禁从 Playwright 控制进程开始计时，会把跨进程调度、动作定位和测试机负载
混入 React 实际反馈时间，因此必须先区分“界面真的晚反馈”和“测试宿主晚收到结果”。

同时，用户可以连续切换不同文件。界面当前允许多个暂存请求并发，必须保证较晚返回的旧状态不会
覆盖其他仍在进行或已经完成的选择。

## 目标

1. 单文件复选框收到真实 click 事件后，勾选状态必须在渲染器下一次可见帧内更新；隐藏窗口门禁上限为 500ms。
2. 暂存命令仍在执行时，当前文件应显示 pending/disabled，但不能延迟第一次视觉反馈。
3. 连续切换多个不同文件后，所有选择必须最终与 Git index 一致，不能因响应乱序丢失先前选择。
4. 切换暂存状态不得重置当前选中的文件或右侧 diff。
5. 性能测试必须使用渲染器内的 `performance.now()` 从真实 click 事件计时，不能用 Playwright 宿主墙钟代替界面反馈时间。
6. Electron 性能测试必须使用独立 `userData` / `sessionData` 配置目录，不与正在使用的开发 Electron 争用 Chromium 缓存。

## 非目标

- 不重做古法 Git 界面或列表样式。
- 不引入整页虚拟化或全局 React 调度改造。
- 不改变 `git add`、取消暂存、提交或丢弃改动的语义。
- 不把磁盘或杀毒软件导致的 Git 命令总耗时伪装成渲染器反馈耗时。

## 验收

- 120 文件 Electron 用例记录 3 次 renderer click-to-paint 时间，全部小于 500ms。
- 三个连续选择在后台请求全部结束后仍全部勾选。
- 右侧选中 diff 在每次切换后保持不变。
- 相关 Node 测试、`pnpm test:quality`、专项 Electron 性能测试和 Windows zip 打包通过。
