# 多窗口流式性能兜底 — 需求

## 目标

在不改变会话语义、不丢消息、不破坏输入焦点、滚动、恢复和持久化的前提下，让 4～6 个长任务同时运行时，Chill Vibe 仍可稳定输入、点击、切换标签和滚动，并避免进入 `BrowserWindow unresponsive`。

本计划不承诺用一次大改“彻底优化”所有性能问题，而是要求每个切片都可证明、可回滚，并且上一切片验证完成后才能进入下一切片。

## 安全契约

- 完整消息、结构化活动、Provider 原生会话和持久化内容不得因 UI 性能兜底而删除、截断、重排或漏写。
- 不得恢复给 reducer 流式提交套 `startTransition` 的旧做法；该做法曾导致活跃 pane 被 React 删除重建和焦点丢失。
- 流结束、停止、报错、恢复、切卡、退出应用前，仍必须无损 flush 对应缓冲区。
- 输入框焦点、中文输入、选区、草稿、附件、Ask User、排队发送、手动停止和恢复流程不得退化。
- 性能兜底只改变“何时刷新”和“当前挂载多少 UI”，不改变 Provider 收到的上下文。
- 每个行为变化必须有独立开关或清晰回滚点；出现数据或交互回归时立即回滚当前切片，不叠加下一项修改。

## 验收标准

### 确定性压力场景

建立隐藏窗口 Electron 压力夹具，至少覆盖：

- 6 个同时处于 `streaming` 的 Codex 卡片；
- 总计约 1000 条消息、900 条以上结构化活动；
- 至少两个当前流式工具组各有 300 条以上 command/tool item；
- 按生产节奏持续注入 delta 和 activity；
- 压力期间循环执行输入、中文文本插入、点击聚焦、滚动、切换 pane tab、展开/折叠和显示更早活动。

### 响应性

- 压力测试期间 `BrowserWindow unresponsive` 和 `render-process-gone` 次数均为 0。
- 主线程不得出现 2 秒以上无心跳；目标最大 stall 小于 500ms。
- 输入到 textarea 值更新的 p95 小于 100ms，点击到聚焦的 p95 小于 150ms。
- 切换 tab、滚动和展开活动不得超时或需要重复操作。
- 5 分钟自动化压力测试通过后，发布候选包还需完成 30 分钟本机隐藏窗口 soak。

### 数据正确性

- 压力前后每张卡的消息 ID、顺序、数量和结构化数据一致；仅允许预期中的新增流消息。
- 当前 UI 隐藏的活动在 Provider 上下文、保存状态和恢复状态中仍完整存在。
- 流完成、手动停止、恢复、应用关闭时不会遗留未 flush 的 delta/activity。
- 交错的多个 assistant item 继续按 `messageId` 独立合并，不串消息、不覆盖。
- 同一条流消息因重连或事件重放被再次提交时，按消息 ID 幂等处理，不在内存或持久化状态中产生重复消息。
- 重新加载后，排队发送、草稿、附件、sessionId、streamId 恢复语义保持不变。

### 交互与视觉

- 活跃流式工具组默认只挂载有界尾部，但标题显示完整总数。
- 显示更早活动按有界批次进行；新活动到达后最新项始终可见。
- 折叠组保持 header-only；普通短会话不出现无意义的性能提示。
- light/dark、桌面和窄视口都不引入新的布局跳动、焦点样式或滚动锚点变化。

## 非目标

- 第一阶段不改 Provider 协议、不自动执行 `/compact`、不改持久化 schema。
- Windows 不得继续无条件关闭硬件加速；必须保留 `CHILL_VIBE_DISABLE_HARDWARE_ACCELERATION=1` 作为兼容性回退。
- 不在缺少压力测试证据时引入整页虚拟列表、全局状态库迁移或 React 架构重写。
- 不为了跑分降低消息完整性、隐藏错误或降低恢复可靠性。
