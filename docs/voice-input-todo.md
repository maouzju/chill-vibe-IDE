# Chill Vibe 语音输入 Todo

## 背景

Chill Vibe 的核心工作面是聊天卡片和输入框。语音输入如果要接入，目标不是把产品做成全双工语音助手，而是降低用户写长 prompt、补充修改意见、拆任务、描述命令和口述思路的输入成本。

对这个产品，最佳形态不是“用户一直开着麦克风和 IDE 对话”，而是：

`按住说话 -> 实时转写 -> 写入 composer -> 用户确认后发送`

这条链路有几个原则：

- 语音只负责把话变成草稿，不直接替代聊天发送。
- IDE 场景里误识别成本很高，所以必须可编辑、可撤回、可理解。
- 不能要求所有用户先去买语音 API；默认就该有免费可用模式。

## 产品结论

### 默认方案

- 默认交互：`push-to-talk`
- 默认识别引擎：本地 `whisper.cpp`
- 默认行为：只把转写文本写进输入框，不自动发送
- 默认隐私策略：不持久化原始音频，只保留转写文本和必要的轻量状态

### 可选增强方案

- 高精度云端模式支持用户自带 key
- 全球优先：`OpenAI gpt-4o-transcribe`
- 中国大陆优先：`火山引擎 / 豆包语音识别`
- 语音识别 provider 和聊天模型 provider 解耦

这意味着用户可以：

- 语音识别用本地 `whisper.cpp`
- 或者语音识别走 OpenAI / 豆包
- 同时聊天模型仍然继续用 Codex / Claude

## 为什么默认选本地 whisper.cpp

### 选它的原因

- 本地离线可用，不要求用户先买 API
- Electron 产品里可以作为 sidecar 进程接入，不必一开始就做原生 Node 绑定
- 社区成熟度够高，资料和现成集成方式比纯小众离线识别器更好找
- 对“把语音变成草稿”这个目标已经够用

### 不把它神化

- `whisper.cpp` 的主要成本不在接入代码，而在模型文件和机器性能
- 它不是当前识别效果最强的方案，只是“本地默认模式”的最稳选择
- 中文、英文、文件名、命令名、变量名混说时，仍然需要后处理和术语表来补

## 非目标

- 不做持续监听的默认模式
- 不做第一阶段就自动发送
- 不做浏览器 `Web Speech API` 主方案
- 不做第一阶段 TTS 或 AI 语音回复
- 不把语音识别直接并进 Codex / Claude CLI 链路

## 实现策略

### 架构拆分

- Renderer 负责麦克风采集、录音态 UI、草稿合并
- Electron main 负责权限、sidecar 生命周期、日志和异常兜底
- 本地 STT provider 先接 `whisper.cpp`
- 云端 STT provider 后续抽象成统一接口
- 最终文本回到现有 composer 草稿状态，而不是伪装成消息

### 统一接口

建议把语音模块抽象成一套独立接口：

- `startCapture`
- `stopCapture`
- `cancelCapture`
- `streamPartial`
- `commitFinalTranscript`

这样本地和云端都可以走同一套前端交互，不会把 UI 和 provider 绑死。

### 接入方式

第一版不要做 Node 原生绑定，优先用独立 sidecar：

- Electron 启动或首次用到语音时拉起本地 `whisper.cpp` 进程
- 优先复用 `whisper-server` 或等价本地 HTTP / streaming 服务
- 应用只负责喂音频和收文本

这样做的好处：

- 跨平台打包简单得多
- 调试容易
- 崩了可以单独重启
- 以后切 `faster-whisper` 或别的本地引擎时也更容易替换

## 体验要求

- 用户在普通聊天卡片里按住麦克风，说一句中文或英文，必须能在一个闭环内看到文本进输入框
- 默认不自动发送
- 切换卡片、切换主题、重启应用后，语音模块不能把输入框状态搞乱
- 在低配机器上，即使识别变慢，UI 也不能卡死
- 错误必须可理解、可恢复，不允许静默失败

## 风险

- `whisper.cpp` 最大的问题是模型文件和机器性能，不是“怎么调一个 npm 包”
- 本地模式不能承诺永久免费且效果最佳
- 云端模式虽然效果更高，但会引入网络、计费和隐私说明成本
- IDE 场景下误识别代价高，所以产品要优先保证可编辑和可撤销，而不是盲目追求自动发送

## 推荐的首版拍板

如果现在就要拍板，我建议首版范围定成这样：

- 默认：本地 `whisper.cpp`
- 交互：`push-to-talk`
- 输出：只写 composer，不自动发送
- 打包：不内置模型，首次使用按需下载
- Provider：先做本地，第二阶段再补 OpenAI / 豆包

这套方案对 Chill Vibe 最稳，也最符合这个产品“本地优先、AI 聊天为核心、降低配置门槛”的方向。
